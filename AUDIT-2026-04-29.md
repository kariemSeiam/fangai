## BUGS

### Critical

**`adapters.ts:349` ‚Äî AiderAdapter hardcodes `'--message', 'CMD'` placeholder**
Severity: **CRITICAL**
Every Aider task sends literal "CMD" as the message instead of the real task. ALL aider tasks are broken.
Fix: Remove `'--message', 'CMD'` from `buildArgs`. Pass `task.message` as positional arg.

**`adapters.ts:464-479` ‚Äî GeminiAdapter never emits `completed`**
Severity: **CRITICAL**
No `completed` detection in `parseLine`. Gemini tasks can never finish cleanly ‚Äî they always time out.
Fix: Add detection for Gemini's completion event type in `parseLine`.

**`server.ts:140-214` ‚Äî `executePersistent` returns before Pi task completes**
Severity: **CRITICAL**
`executePersistent` doesn't return a promise that waits for completion. Pi tasks likely return immediately before the task actually runs.
Fix: Wrap in `new Promise<void>` that resolves on completion/error/timeout.

### High

**`server.ts:147-150` ‚Äî Race on `this.persistent` initialization**
Severity: **HIGH**
Two concurrent calls both see `this.persistent === null`, both create a `PersistentProcess`, second overwrites first. First process leaked.
Fix: Use a lazy-init guard:
```typescript
private persistentInit: Promise<PersistentProcess> | null = null;
private async getPersistent(task, config): Promise<PersistentProcess> {
  return (this.persistentInit ??= (async () => { const p = new PersistentProcess(...); await p.ensure(); return p; })());
}
```

**`server.ts:173-214` ‚Äî Concurrent `write()` calls interleave Pi session**
Severity: **HIGH**
Two A2A requests hitting Pi simultaneously ‚Äî both call `this.persistent.write()` back-to-back. Pi receives two prompts in rapid succession, may start processing B while responding to A. Output routing sends all output to active task handler, corrupting results.
Fix: `executePersistent` must not call `write()` until previous task completes. A write queue in `PersistentProcess`.

**`adapters.ts:111-115` ‚Äî Pi tool args `JSON.parse` without try-catch**
Severity: **HIGH**
Malformed Pi tool args crash the entire server.
Fix: Wrap `JSON.parse(tc.arguments)` in try-catch.

**`server.ts:225` ‚Äî Processes leak on graceful shutdown**
Severity: **HIGH**
Missing `await this.pm.killAll()`. Spawned processes survive server exit.
Fix: Add `await this.pm.killAll()` to shutdown handler.

### Medium

**`server.ts:293-303` ‚Äî Auth middleware blocks agent card discovery**
Severity: **MEDIUM**
When `config.apiKey` is set, auth runs before ALL routes including `/.well-known/agent-card.json`. Per A2A spec, agent cards must be publicly discoverable.
Fix: Exclude `/.well-known/agent-card.json` and `/health` from auth middleware.

**`client.ts:39-175` ‚Äî No timeout on `send`/`sendStream`**
Severity: **MEDIUM**
`health()` uses `AbortSignal.timeout(3000)` but `send`/`sendStream` have no timeout. Hung server = infinite block.
Fix: Accept optional `timeout` in opts, apply `AbortSignal.timeout(ms)`.

### Low

**`core.ts:239-248` ‚Äî `setLineHandler` inserts task into queue in BOTH branches**
Severity: **LOW**
Both branches push to `taskQueue`. Queue conflates "active" and "pending" confusingly. `advanceQueue` never removes completed tasks before advancing.
Fix: Treat `taskQueue` as pending-only. Don't add initial task to queue.

**`client.ts:82` ‚Äî SSE multi-line `data:` fields silently truncated**
Severity: **LOW**
Parser takes only first `data:` line, discards rest per SSE multi-line spec. Unlikely to trigger with real A2A streams but spec non-compliant.

## ARCHITECTURE ISSUES

1. **Persistent mode has no serialization layer.** `PersistentProcess` queues handlers but `executePersistent` writes immediately without waiting for queue drain. Write and queue are decoupled.

2. **`BridgeExecutor` holds adapter and config at construction time.** Fixed at startup. No hot-reload path.

3. **`InMemoryTaskStore` = zero durability.** Server restart loses all task state. No eviction policy ‚Äî store grows unbounded under high load.

4. **Agent card URL hardcoded to `localhost`.** Wrong behind reverse proxy, Docker port forwarding, or remote host. Should be configurable via `publicUrl`.

5. **`splitCli` is a hand-rolled arg parser** (server.ts:238-249). Doesn't handle escaped quotes or complex cases. Edge case fragility.

## ADAPTER ISSUES

### Pi (Tier 1, persistent)
- Both `turn_end` (line 152) and `agent_end` (line 182) emit `completed`. Second dropped by `settled` flag but fires twice.
- `message_end` contains full accumulated text but is dropped (`return []`). No fallback recovery if streaming deltas are lost.
- `extension_ui_request` double-handled: adapter drops it, `PersistentProcess` auto-responds. Correct behavior but misleading comment.

### Claude (Tier 1, oneshot)
- `buildArgs` hardcodes `--max-turns 10` (line 227). Should come from config.
- Non-JSON stdout appended with `\n` (line 243): adds extra newlines for multi-line error banners.
- `result` with `is_error: true` emits `failed` status, but `executeOneshot` only handles `completed` ‚Äî no handler for `failed` state. Falls through to `onExit`. Functional but adds latency.

### Cursor (Tier 1, oneshot)
- `binary = 'agent'` (line 282) ‚Äî extremely generic name. High false-positive risk.
- `parseLine` handles `result` ‚Üí `completed` (line 316) but Cursor's actual format may use `message_stop`. Needs verification.
- Not exported from `index.ts`. Inaccessible to external consumers.

### Aider (Tier 3, oneshot)
- **`buildArgs` returns `['--message', 'CMD', ...]`** ‚Äî literal placeholder "CMD". All tasks broken.
- `parseLine` passthrough: git diff headers, progress bars, model loading all shown as agent output.
- No `completed` signal. Relies entirely on process exit.

### Codex (Tier 1, oneshot)
- No handling for `item.error` event type. Errors treated as text output.
- Positional message via stdin may conflict with `--json` flag. Unverified.

### Gemini (Tier 2, oneshot)
- **Never emits `completed`** ‚Äî see BUGS/CRITICAL.
- `formatInput` uses hardcoded `id: 1` for JSON-RPC. Session correlation impossible.
- `sessionId` instance variable (line 462) is dead state ‚Äî set but never read. Singleton means concurrent requests corrupt each other.
- Assumes Gemini auto-starts a session. Unverified.

### OpenCode (Tier 2, oneshot)
- `buildArgs` appends `task.message` directly as positional arg (line 509). Multi-word messages may not work depending on CLI design. Needs `--` terminator.
- `parseLine` computes `const text = obj.text || obj.content || obj.part?.text || ''` (line 521) but `type === 'content'` uses `obj.text || ''` instead of computed `text` (line 524). Inconsistency.

### Generic (Tier 3, oneshot)
- Never emits `completed` by design. Always relies on process exit code.
- `detect()` splits `cliCommand` by space (line 566) ‚Äî same fragility as `splitCli`.

## SECURITY ISSUES

1. **CORS `*` with no API key = completely open endpoint** (server.ts:284). Fine for local; dangerous if exposed to internet.
2. **OpenCode message as positional arg** (adapters.ts:509) ‚Äî no shell injection since spawn doesn't use shell, but user can pass `--flag` prefixed strings. Fix: use `--` terminator.
3. **Agent card leaks internal server config** (server.ts:258-267). If agent name includes username/path, leaks PII.
4. **YAML config parsed without schema validation** (cli.ts:70-71). Malformed `fang.yaml` could pass `cli: 'rm -rf /'` (low threat model for user-controlled tool).
5. **No request body size limit on subprocess forwarding.** `express.json({ limit: '10mb' })` limits raw body but 9MB payload forwarded entirely to subprocess. Unbounded memory per request.
